# RPM Repository Setup Instructions

This guide explains how to set up and host a YUM/DNF RPM repository on GitHub for the hotaisle-cli package. It mirrors the Debian setup in [APT_REPO_SETUP.md](APT_REPO_SETUP.md) and reuses the same GPG signing key.

## Repository Structure

Create a new GitHub repository `hotaisle/rpm-repo` with the following structure:

```
rpm-repo/
├── gpg.key                # Public signing key, committed once during setup
├── packages/              # rpm packages go here
│   └── (rpm packages go here)
└── repodata/              # Generated by createrepo_c
    ├── repomd.xml
    ├── repomd.xml.asc     # Detached signature of repomd.xml
    └── ...
```

The release workflow regenerates `repodata/` and the detached signature on every release. The `gpg.key` public key and the repository itself are created once during setup and are not overwritten by the workflow, so any change to the key is visible as its own commit.

## Initial Setup Steps

### 1. Create the RPM Repository

```bash
# Create new repository
mkdir rpm-repo
cd rpm-repo

# Create directory structure
mkdir -p packages

# Initialize git
git init
git remote add origin git@github.com:hotaisle/rpm-repo.git
```

### 2. GPG Key for Package Signing

The RPM repository is signed with the **same GPG key** used for the APT
repository. If you have already completed
[APT_REPO_SETUP.md](APT_REPO_SETUP.md), no new key is required. Otherwise,
generate one as described in that guide.

Add the public key to the repository as `gpg.key` so users can import it. The
release workflow does not write this file, so committing it directly keeps any
future key change visible in the git history:

```bash
cd rpm-repo
gpg --armor --export YOUR_KEY_ID > gpg.key
git add gpg.key
git commit -m "Add public GPG key"
git push origin main
```

### 3. Configure GitHub Secrets

These secrets must exist on the `hotaisle/hotaisle-cli` repository (they are the
same ones used by the APT release steps):

- `APT_GPG_PRIVATE_KEY`: Content of the armored private key
- `APT_GPG_PASSPHRASE`: Passphrase for the private key
- `APT_GPG_KEY_ID`: The GPG key ID (format: `ABCD1234EFGH5678`)
- `PERSONAL_ACCESS_TOKEN`: Token with push access to `hotaisle/rpm-repo`

### 4. Enable GitHub Pages for the RPM Repository

1. Go to the `hotaisle/rpm-repo` repository settings
2. Navigate to Pages section
3. Set Source to "Deploy from a branch"
4. Select branch: `main`, folder: `/ (root)`
5. Save

Your RPM repository will be available at: `https://hotaisle.github.io/rpm-repo`

### 5. How Releases Publish RPMs

On each merged release PR, `.github/workflows/release.yml`:

1. Builds the signed binaries and `.rpm` packages (`just release`).
2. Checks out `hotaisle/rpm-repo`.
3. Copies `dist-pkg/*.rpm` into `packages/`.
4. Runs `createrepo_c --update` to regenerate `repodata/`.
5. Signs `repodata/repomd.xml` with the GPG key (detached and armored, the RPM
   equivalent of apt's signed `InRelease`).
6. Commits and pushes back to `hotaisle/rpm-repo`.

The committed `gpg.key` is left untouched by the workflow, so it only changes
when a maintainer updates it directly.

Trust flows from the signed `repomd.xml` to the metadata checksums to the
package checksums, exactly like apt's signed `Release` file. This is why the
user `.repo` config below uses `repo_gpgcheck=1`.

## User Installation Instructions

### For End Users

Create a file `/etc/yum.repos.d/hotaisle.repo`:

```bash
sudo tee /etc/yum.repos.d/hotaisle.repo > /dev/null <<'EOF'
[hotaisle]
name=Hot Aisle
baseurl=https://hotaisle.github.io/rpm-repo
enabled=1
repo_gpgcheck=1
gpgcheck=0
gpgkey=https://hotaisle.github.io/rpm-repo/gpg.key
EOF

# Install
sudo dnf install hotaisle
```
